# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

boringproxy is a reverse tunnel / reverse proxy manager written in Go. A single binary runs in two roles:

- **server**: runs on a machine with a public IP, terminates HTTP/HTTPS/TLS-SNI on ports 80/443, manages an embedded web UI + JSON API, and issues short-lived SSH keypairs so agents can open reverse tunnels (via the local `sshd`, using `ssh.Listen` on the agent side). boringproxy never runs its own SSH listener -- "the local `sshd`" is the host machine's real system SSH daemon, which the server only manages `~/.ssh/authorized_keys` entries for (`tunnel_manager.go`'s `addToAuthorizedKeys`); in Selfie Proxy specifically this is possible because `selfieproxy-reverseproxy` runs `network_mode: host` and bind-mounts `/root/.ssh` from the host (see root `CLAUDE.md`). `-ssh-server-port` (default `22`, `Config.SshServerPort`) is the port value stamped onto every `Tunnel.ServerPort` an agent is told to dial (`api.go`'s tunnel-list handler, re-stamped on every poll) -- it does not configure the host's sshd itself, so changing it requires separately reconfiguring that sshd's own `Port` directive to match, or agents will fail to connect. No flag or config exists on the agent side for this -- the agent always dials whatever `ServerAddress`/`ServerPort` the server's `/api/tunnels` response carries per tunnel (`agent.go`'s `BoreTunnel`), so a value change propagates to already-running agents on their next poll with no redeploy needed there. Not currently exposed through this repo's `.env`/docker-compose.yaml directly, but see `-stealth-mode` below (`STEALTH_MODE` in root `CLAUDE.md`), which overrides the effective value to `443`.

  `-stealth-mode` (default `false`, `Config.StealthMode`) disguises the agent's SSH reverse tunnel as HTTPS on port 443 instead of dialing `-ssh-server-port` (22) directly, for agents on networks that block outbound SSH but allow 443. It forces the effective `SshServerPort` to `443` and turns on an ALPN-gated dispatch branch in `Server.handleConnection` (`boringproxy.go`) that sits ahead of the normal SNI/tunnel lookup: a connection addressed to the admin domain *and* offering the custom ALPN protocol ID `selfieproxy-ssh` (the const `stealthSshAlpn`) in its ClientHello is real-TLS-terminated (reusing `tls_proxy.go`'s `ProxyTcp`, the same machinery `server-tls` tunnels already use) and the decrypted plaintext is piped to `127.0.0.1:22` (`hostSshdPort`, hardcoded -- the real, unconfigurable host sshd port, deliberately decoupled from the public-facing 443 under stealth mode). Ordinary HTTPS traffic to the admin domain (portal/agent API calls, browser requests -- ALPN `h2`/`http/1.1`) and ACME's own `acme-tls/1` challenge traffic never carry that ALPN value, so both fall through to the existing logic completely unaffected. No new subdomain, DNS record, or cert is needed: agents already dial the admin domain for SSH (`api.go`'s `sshServerAddr := a.db.GetAdminDomain()`), and its cert is already eagerly managed by certmagic at startup. On the agent side, `BoreTunnel` (`agent.go`) wraps its dial in a real `tls.Dial` (offering only the `selfieproxy-ssh` ALPN, `InsecureSkipVerify: true`) instead of a plain `net.Dial` whenever the tunnel's `SshTls` field is set (re-stamped from `Config.StealthMode` alongside `ServerPort` on every poll, same propagation mechanism) -- `InsecureSkipVerify` is safe here because TLS is only a disguise layer to defeat DPI-based port/content filtering, not the trust boundary; the SSH handshake immediately inside it still authenticates with the tunnel's own short-lived ephemeral keypair, unchanged.
- **agent**: runs on the machine hosting the actual service being exposed, polls the server's `/api/tunnels` for its assigned tunnels, and opens an SSH reverse tunnel (`client.Listen`) for each one, then proxies HTTP/TCP through it to the local service.

The root package `boringproxy` (module `github.com/boringproxy/boringproxy`) contains all core logic; `cmd/boringproxy` is the CLI entrypoint (`main.go`) that parses subcommands and calls into the package.

## Build

Requires Go 1.26+ (bumped from 1.17 when the embedded OIDC Relying Party was added â€”
`github.com/coreos/go-oidc/v3` and `golang.org/x/oauth2` both need a modern toolchain; see
`Dockerfile`'s `golang:1.26-alpine` builder stage).

```bash
# One-time: logo.png must exist at repo root (embedded via go:embed in ui_handler.go)
./scripts/generate_logo.sh   # requires inkscape; logo.png is already checked in, so usually not needed

cd cmd/boringproxy
go build
# or with version info:
go build -ldflags "-X main.Version=$(git describe --tags)"

# allow binding to low ports (80/443) without root:
sudo setcap cap_net_bind_service=+ep boringproxy
```

`scripts/build.sh` does the build + setcap in one step. `scripts/build_release.sh` cross-compiles for all supported platforms (see `.goreleaser.yml`) and is used for tagged releases.

There are no test files in this repo (`*_test.go` does not exist) and no lint config â€” `go build ./...` and `go vet ./...` are the available correctness checks.

## Running

```bash
./boringproxy server                                                    # start a server
./boringproxy agent -server example.com -secret <SECRET> -agent-name <NAME> -user <USER>
./boringproxy tuntls -server example.com                                # raw TLS tunnel over stdin/stdout, ported from anderspitman/tuntls
```

Key server flags: `-admin-domain`, `-portal-domain`/`-portal-port`, `-sso-domain`/`-sso-port`, `-oidc-issuer`/`-oidc-client-id`/`-oidc-client-secret`, `-sso-idle-minutes`/`-sso-max-minutes` (default `30`/`600`, i.e. 30 minutes idle / 10 hours absolute -- see the OIDC section below), `-db-dir`, `-cert-dir`, `-http-port`/`-https-port`, `-allow-http`, `-acme-email`, `-acme-use-staging`, `-ssh-server-port` (default `22` -- see "What this is" above for why this only changes what agents are told, not the host sshd's own listening port), `-stealth-mode` (default `false` -- see "What this is" above), `-debug` (logs every request -- timestamp, remote IP, method, host, path -- to stdout; off by default since every agent poll and portal auto-refresh would otherwise log a line).

Key agent flags: `-server`, `-secret`, `-agent-name`, `-user`, `-poll-interval-ms` (min 100ms, 0 disables polling), `-dns-server`.

Upstream boringproxy has a `-behind-proxy` flag on both server and agent that appends to (rather than overwrites) a downstream-supplied `X-Forwarded-For` -- this fork removes it entirely (`http_proxy.go`'s `proxyRequest`/`proxyWebsocket`, `boringproxy.go`, `agent.go`, `cmd/boringproxy/main.go`) rather than keep it default-off. Trusting a client-supplied XFF value is only safe if the previous hop is fully controlled and known to have sanitized it -- a guarantee this deployment can't make (a self-hosted reverse-tunnel engine has no fixed, trusted "upstream LB" of its own). `X-Forwarded-For` is now unconditionally set to the real, non-spoofable immediate peer IP in every proxying path (plain HTTP and websocket), same as the flag's default (and only ever used) behavior. This was investigated after a report that the agent-side half of this flag (`agent.go`'s old `behindProxy` field) was already dead code for Selfie Proxy specifically -- it only ever fired for `Tunnel.TlsTermination == "client"`, a value `selfieproxy-portal`'s `TunnelMapper` never produces (it only emits `"server"`/`"client-tls"`/`"server-tls"`/`"passthrough"`) -- but removing the flag entirely closes off the underlying footgun rather than leaving unreachable-but-present machinery that a future change could silently reactivate.

On first run with no admin domain, the server prompts interactively (or via the TakingNames.io integration) to set one; on first run with no users, it creates an `admin` user and a corresponding root token, printable via `-print-login`.

## Architecture

### Connection flow (server side, `boringproxy.go`)

A single raw `net.Listener` on the HTTPS port feeds `Server.handleConnection`, which peeks the TLS ClientHello (`sni.go`) to read the SNI hostname *without* consuming the connection, then dispatches based on the tunnel's `TlsTermination` mode looked up by SNI hostname in the DB. Ahead of that DB lookup, one more check runs first when `-stealth-mode` is on: `peekClientHello` already surfaces `tls.ClientHelloInfo.SupportedProtos` (the ClientHello's ALPN list) for free, so a connection to the admin domain offering the custom ALPN ID `selfieproxy-ssh` is diverted straight to the real host sshd (`127.0.0.1:22`) via `ProxyTcp`, before the tunnel lookup ever runs -- see "What this is" above for the full stealth-mode design.

- `client` / `passthrough` / `client-tls`: raw bytes are piped straight through to the client's tunnel port (`passthroughRequest`) â€” TLS is terminated by the client (or not at all).
- `server-tls`: the server terminates TLS itself and forwards decrypted TCP to the client (`tls_proxy.go`'s `ProxyTcp`).
- anything else (including admin domain / plain HTTP tunnels, and a domain with no `Tunnel` record at all â€” see the self-signed-cert paragraph below for how that reaches here without failing the TLS handshake first): handed to `PassthroughListener`, which feeds a normal `http.Serve` loop wired to certmagic for ACME certs, and requests are routed by `Host` header â€” either to the admin API/web UI, or proxied to the tunnel's local port (`http_proxy.go`'s `proxyRequest`), or, if `db.GetTunnel(hostDomain)` finds nothing at all, a 404 page (see `writeHtmlError` below) saying the domain isn't configured. Since Network Services are always `passthrough` (see `TunnelMapper`) and `server-tls`-terminated tunnels are handled above without ever reaching `proxyRequest`, every ordinary tunnel this branch proxies is necessarily a Web Application (or the hidden This Server homelab's Local Websites) â€” which is what lets `proxyRequest`'s `upstreamErrorAgentUnreachable` mode below assume that wording unconditionally.

`proxyRequest`'s `errorMode` parameter (`upstreamErrorMode`) governs what a failed `httpClient.Do` to `address:port` reports back, since the same function serves audiences with very different needs: `upstreamErrorDefault` (the agent's own call proxying to the real homelab backend, see below) reports the raw error with a 502; `upstreamErrorStartingUp` (the portal/sso/console synthetic-tunnel calls below) reports 200 with "Selfie Proxy is starting, please retry later", since a boot-time `connection reset by peer`/`EOF` against a Java service that hasn't opened its listener yet isn't a real failure, just a race between this proxy coming up and that service finishing its own startup; `upstreamErrorAgentUnreachable` (the ordinary-tunnel call above) reports 404 with one of two small standalone HTML error pages (`writeHtmlError`, `error_page.go`), distinguished by inspecting `err` itself (`errors.As` for a `*net.OpError` with `Op == "dial"`) rather than by any state this server tracks: a dial failure means nothing is listening on `127.0.0.1:<tunnel port>` at all, i.e. the agent's own SSH reverse-tunnel bore for this app is gone (agent process down, or its SSH connection dropped) â€” "404 - Agent Not Found" / "the Selfie Proxy agent is disconnected" / a pointer to re-check the homelab's name/secret in the agent's `docker-compose.yaml`; anything else reaching this branch is a mid-response EOF (the dial to the tunnel port succeeded -- the agent is connected -- but it wrote no response at all, because its own dial to the real backend failed, see `tls_proxy.go`'s `handleConnection` below) â€” "404 - Server Not Found" / "the agent could not connect to the server in your homelab" / a pointer to check the app's "Address in the homelab" settings. Neither page ever includes the homelab's internal address/port. `error_page.go`'s `writeHtmlError`/`errorPageHead` are the single shared template (title + font-family `<style>`) behind every such page this proxy serves directly, including the no-tunnel-at-all 404 below â€” only the `<h1>`/body text differ per call site.

A domain with no `Tunnel` record at all used to fail the TLS handshake itself before any of this could run: `certConfig.GetCertificate` has no managed cert for it (nothing ever called `ManageSync` on it) and the old self-signed fallback only covered domains with a *pending* real cert, so `crypto/tls` sent an `internal_error` alert (`SSL_ERROR_INTERNAL_ERROR_ALERT` in Firefox, `ERR_CERT_AUTHORITY_INVALID` in Chrome) with no way to reach the HTTP layer at all. `self_signed_cert.go`'s `withSelfSignedFallback` now takes a `certFallback` decision instead of a plain pending/not-pending bool, but it still only has two outcomes: `certFallbackSelfSigned` (known tunnel with a pending real cert, *or* no tunnel at all) serves `SelfSignedCertProvider`'s per-domain cached self-signed cert; `certFallbackNone` propagates the real error. Both self-signed cases share the same cache â€” an earlier version tried generating a fresh, uncached cert per handshake specifically for the no-tunnel case (reasoning that the domain name there is fully visitor/attacker-controlled, via the wildcard DNS record that makes any subdomain resolve here, so caching it would let a scanner force unbounded memory growth) but that broke the actual feature: a browser's "proceed anyway" exception is keyed to the exact certificate bytes, so a new key/serial on every single connection made it re-prompt on the very next request (reload, sub-resource, anything) instead of ever sticking â€” confirmed live as `ERR_CERT_AUTHORITY_INVALID` persisting after clicking through. `SelfSignedCertProvider` now caches unconditionally but caps itself at `selfSignedCertCacheLimit` (1000), evicting the oldest entry (FIFO) once full â€” bounding the same memory-growth concern without breaking repeat visits, since legitimate use (a handful of tunnels with pending certs, plus normal traffic hitting a few misconfigured/typo'd domains) never gets close to the cap. The tradeoff that remains: the visitor still has to click through the self-signed-certificate browser warning once per domain (unavoidable â€” there's no legitimate cert for a domain nobody configured), but it now actually stays clicked-through, and past it they see the domain-not-configured 404 page above instead of a hard connection failure.

### Agent tunnel lifecycle (`agent.go`)

`Agent.Run` registers the agent with the server (`POST /api/agents/`), then loops: `PollTunnels` does a conditional GET against `/api/tunnels` (ETag-based, ETag is an md5 of the tunnel list, so unchanged sets short-circuit) and calls `SyncTunnels`, which diffs the new tunnel map against the current one, starting (`BoreTunnel`) any new/changed tunnels via SSH and cancelling+removing (via per-tunnel `context.CancelFunc`) any that were deleted server-side. Each `BoreTunnel` dials the server over SSH using a private key issued by the server for that tunnel, opens a remote listener, and locally proxies each accepted connection to the agent's actual service (HTTP reverse-proxying for `client` TLS termination, raw/TLS TCP proxy otherwise -- the raw/TLS path (`tls_proxy.go`'s `ProxyTcp`/`handleConnection`) sends `tunnel.Domain` as the TLS ServerName/SNI on the upstream dial, since a homelab HTTPS backend doing SNI-based vhost routing needs the public FQDN, not the tunnel's internal `ClientAddress`, which is often a bare IP or LAN-only hostname). In practice every `TunnelMapper`-produced Web Application tunnel is `server`-terminated, so it's this raw/TLS path, not the `client`-mode HTTP branch, that actually handles a homelab backend being unreachable: `handleConnection`'s dial failure just logs (e.g. `dial tcp 192.168.1.11:8006: connect: connection refused`) and closes the tunnel connection with no response written at all, which is exactly what the server's own `proxyRequest` sees as an `httpClient.Do` error (an EOF reading a response that was never sent) -- see the Connection flow section above for how that becomes a 404 rather than echoing the agent's local dial error (and the address it names) back to the browser.

### Core types (`database.go`)

`Database` is a single JSON file (`boringproxy_db.json` in `-db-dir`) guarded by one mutex, persisted synchronously on every mutation (`persist()` calls `saveJson`) â€” there's no transaction/locking across read-then-write API sequences (a known TODO). Top-level entities: `Tokens` (map of token string -> `TokenData{Owner, Agent}`), `Tunnels` (map of domain -> `Tunnel`), `Users` (map of username -> `User{IsAdmin, Agents}`). A `Tunnel` carries both the routing/auth config (domain, TLS termination mode, optional basic-auth, owner) and the SSH connection details for that tunnel (per-tunnel keypair, tunnel port, allowed external TCP).

### Authorization model

Every access token maps to a `TokenData{Owner, Agent}`. If `Agent` is set, the token is scoped to one agent and can only be used to poll `/api/tunnels`/`/api/agents` for that agent â€” not to manage tunnels/users/tokens via the web UI or general API. Otherwise the token acts as its `Owner` user, and `User.IsAdmin` gates cross-user operations (creating tunnels/tokens for other users, managing all users). This scoped-token mechanism is also how Selfie Proxy's per-agent secrets work: an agent-scoped token is minted only for an agent name already registered under a user (`Api.CreateToken` refuses otherwise), so an agent's "secret" is just its own restricted access token. This logic is duplicated between `api.go` (JSON API, used by agents and directly) and `ui_handler.go` (server-rendered web UI, which calls into the same `Api` methods) â€” when changing authorization rules, both call sites matter, though several handlers have `// TODO: handle security checks in api` markers indicating the check still lives in `ui_handler.go` rather than `api.go`.

### Web UI (`ui_handler.go` + `templates/*.tmpl`) -- disabled in Selfie Proxy

Server-rendered via `html/template`, templates embedded with `go:embed`. `WebUiHandler.handleWebUiRequest` is a single big switch over `r.URL.Path` (not a router/mux) sitting behind the same token-cookie/`access_token` auth as the API. Slow operations (tunnel creation, which may block on ACME cert issuance) use a 100ms-timeout + polling `/loading` page pattern with a pending-request map (`pendingRequests`) keyed by a random ID, rather than blocking the initial request indefinitely.

Superseded by Selfie Proxy's own admin portal, this legacy UI is no longer wired up: `boringproxy.go`'s main handler never constructs a `WebUiHandler` and returns 403 for any admin-domain request that isn't under `/api/` or `/rest/` (both of which stay live, since the remote agent depends on `/api/` and selfieproxy-portal's `BoringProxyClient` depends on `/rest/`, both reached over the public admin-domain hostname). The `ui_handler.go`/`templates/` code itself is untouched -- only the routing call site was removed -- so re-enabling it is a one-line change in `boringproxy.go` if ever needed.

### Portal domain (`-portal-domain`/`-portal-port`, `boringproxy.go`)

A second special-cased host, alongside the admin domain: any request whose Host matches `-portal-domain` is reverse-proxied directly to `localhost:<-portal-port>` (via the same `proxyRequest` used for ordinary tunnels, with a synthetic `Tunnel{Domain: portalDomain}` -- no Agent, no SSH bore, no DB Tunnel record). This exists specifically so Selfie Proxy's own admin portal is reachable immediately on server startup, before any agent has ever connected -- the portal itself is where you go to create the first agent and read off its secret, so it can't depend on one already existing. Cert acquisition for this domain happens the same way as the admin domain, via `certConfig.ManageSync` at startup (gated on `autoCerts`). The portal domain is now always gated behind single sign on (see below) -- the portal itself has no login left. Called with `proxyRequest`'s `upstreamErrorStartingUp` mode (see Connection flow above), same as `-sso-domain`/`-console-domain` below -- selfieproxy-portal is a separate container that can take a few seconds to finish its own Spring Boot startup after boringproxy's own listener is already up, and a `connection reset by peer` dialing `localhost:<-portal-port>` during that window is a race to retry, not a real error.

### Single sign on domain (`-sso-domain`/`-sso-port`, `boringproxy.go`)

A third special-cased host, wired identically to the portal domain (same synthetic-tunnel proxy, same eager cert acquisition): reverse-proxies to `selfieproxy-identity-provider`, the bundled OIDC Identity Provider. Never itself gated behind single sign on -- it's the IdP the gate calls out to.

### Console domain (`-console-domain`/`-console-port`, `boringproxy.go`)

A fourth special-cased host, wired identically to the portal domain (same synthetic-tunnel proxy, same eager cert acquisition, same always-gated-behind-single-sign-on treatment): reverse-proxies to `selfieproxy-remote-console`, the browser SSH/RDP/VNC console service (see root `CLAUDE.md`). Like the portal domain, `oidc_auth.go`'s `HandleCallback` rejects a login for this domain if the ID token's `is_admin` claim is present and explicitly `false` -- the browser SSH/RDP/VNC console is Homelab-management tooling, admin-only just like the portal itself, never reachable by a login-only User. Unlike the portal domain, this proxying is a genuinely thin pass-through: `selfieproxy-remote-console` runs `network_mode: host` (it must reach `AllowExternalTcp: false` tunnels at `127.0.0.1:<tunnel_port>` and the `selfieproxy-guacd` container it pairs with, neither of which a Docker bridge-network container can reach), so unlike the portal it needs no Docker bridge-network reachability from boringproxy at all -- both sides are host-networked, and the call is a plain `127.0.0.1:<-console-port>` dial like any other localhost-bound service on this host.

### Embedded OIDC Relying Party (`oidc_auth.go`)

boringproxy embeds its own OIDC client (RP), built from `-oidc-issuer`/`-oidc-client-id`/`-oidc-client-secret` (the last blank for the bundled server, a PKCE-only public client -- see `oidc_auth.go`'s doc comments for the full design). `OidcAuthenticator.RequireAuth` is the single check called from `boringproxy.go`'s main dispatch handler, gating: (1) the portal domain, always, and (2) any ordinary tunnel with `Tunnel.SsoProtected == true` -- which is only ever settable for `"server"`-termination tunnels, the only ones this server HTTP-parses (see Connection flow above). Three token kinds, all stateless/self-verifying HMAC-SHA256 (`{payload}.{sig}`, signing key persisted next to `boringproxy_db.json`): a session cookie (`_selfieproxy_sso`, scoped to the exact `Host` it was issued for), a short-lived relay token (`?sso_token=`, ~60s, used once to bounce the browser from the central `/oidc/callback` on the admin domain back to the original app's own domain before setting that domain's session cookie), and the OIDC `state` param itself (encodes `{domain, return_to, PKCE verifier}`, since there's no server-side session store between `/oidc/authorize` and `/oidc/callback`). The session cookie's lifetime defaults match Keycloak's own defaults (the de facto industry standard) but are configurable: a sliding idle deadline (`ssoClaims.Exp`, refreshed on every request that passes `RequireAuth`, `-sso-idle-minutes`, default 30) capped by an absolute maximum set once at login and never extended by activity (`ssoClaims.MaxExp`, `-sso-max-minutes`, default 600 = 10h) -- both live on `OidcAuthenticator` (`idleTTL`/`maxTTL`, set once in `StartOidcAuth` from the flags) rather than as package constants. Each protected domain (portal, and every exposed app protected with single sign on) tracks this independently, since the cookie itself is host-only and there's no cross-domain *cookie* sharing -- opening a second domain always does a fresh authorization-code round trip through `/oidc/authorize` on the admin domain and the IdP. What makes that round trip actually silent (no re-shown login form) is `selfieproxy-identity-provider`'s own login session (`IdpSessionService`, cookie `selfieproxy_idp_session`, scoped to the IdP's own host and sized by the same `-sso-idle-minutes`/`-sso-max-minutes`): `AuthorizeController.startNew` checks it before falling back to `/login`, so logging in anywhere (portal or any app) makes every other domain's next round trip silent too, symmetrically -- not just portal-first. Explicit logout (`/oidc/logout` -> `/logged-out`) also ends that IdP session, so it doesn't silently resurrect logins for domains not yet visited. A cookie signed before these fields existed unmarshals with `MaxExp == 0` and is rejected -- a one-time forced re-login on rollout, not a crash. `/oidc/authorize` and `/oidc/callback` are carve-outs on the admin domain, alongside `/api/`/`/rest/`.

`HandleCallback` also enforces the split between Selfie Proxy's own admin and its non-admin Users
(see `selfieproxy-identity-provider`'s `UserStore`/Users pages): once the verified ID token is in
hand, a login for the portal domain specifically (`OidcAuthenticator.portalDomain`, threaded
through from `StartOidcAuth`'s `*portalDomain` argument -- not checked for any other domain
protected with single sign on) is 403'd if the token's `is_admin` claim is present and explicitly `false`, before any
relay token or session cookie is issued. An *absent* `is_admin` claim -- which is what every IdP
other than the bundled `selfieproxy-identity-provider` produces, since it's a non-standard claim --
is treated as permissive, so this whole restriction is naturally scoped to the bundled IdP without
`oidc_auth.go` needing to know which IdP is actually configured.
`/oidc/logout` is a separate carve-out checked first, on *every* domain rather than just the
admin domain (`boringproxy.go`'s dispatch, ahead of the admin/portal/sso/tunnel branching) --
since the session cookie it clears is host-only, logout has to run on whatever domain the
browser is currently on. `HandleLogout` (`oidc_auth.go`) doesn't depend on `oidcAuthHolder`
being populated, unlike authorize/callback, since clearing a cookie needs no IdP round trip; it
redirects to `/logged-out` on `-sso-domain` (the bundled IdP, always reachable regardless of
whether `-oidc-issuer` points elsewhere) for a confirmation page, then a `return_to` link back
into the app that triggered the logout.

OIDC discovery against `-oidc-issuer` runs in a background goroutine with retry/backoff (`StartOidcAuth`), never synchronously at startup -- `oidcAuthHolder` (an `atomic.Pointer[OidcAuthenticator]`) starts nil and every dispatch-path caller treats nil as "Selfieproxy is starting... please retry shortly" (503), not "single sign on disabled". This matters because boringproxy's own discovery call is an HTTP request that has to reach the single sign on server *through boringproxy's own listener* (the `-sso-domain` carve-out above) -- doing discovery synchronously before that listener starts would deadlock, since the listener can't be up yet during discovery's own call site in `main()`. `selfieproxy-identity-provider` boots concurrently with boringproxy (`docker-compose.yaml` has no `depends_on` between them -- both only wait on the `check-prerequisites` preflight), rather than waiting for boringproxy to report healthy first, so it's typically most of the way through its own (independent) Spring Boot startup by the time boringproxy's listener comes up. The retry backoff (`ssoDiscoveryMinWait`/`ssoDiscoveryMaxWait`, 250ms-3s) is deliberately short, since this is a same-host loopback call with no external rate limit to respect -- it exists only to bound the polling rate, not to protect a remote service.

### TakingNames.io / namedrop integration

Optional DNS automation via `github.com/takingnames/namedrop-go`: lets a user link a domain through TakingNames.io instead of manually pointing DNS at the server, handled via the `/namedrop/callback` route in `boringproxy.go`'s main handler and `Config.namedropClient`.
